# Solución sugerida para estrategia de ramas
Lista para evaluar antes de implementarla.

---

## Objetivo
Tener flujo simple, auditable y rápido para el examen, minimizando riesgo de romper producción.

---

## Estrategia recomendada
- Rama **main**
- Rama **develop**
- Ramas **feature/***
- Ramas **hotfix/*** opcionales para urgencias desde main

---

## Reglas de uso

### main
- Solo código listo para entregar.
- Siempre protegida.
- Solo merge vía Pull Request.
- Requiere CI en verde.
- Cada merge a main debe dejar un tag de versión.

### develop
- Rama de integración diaria.
- Recibe merge de feature/*.
- También protegida, pero con reglas un poco menos estrictas que main.
- Debe mantenerse estable para staging.

### feature/*
- Se crean desde develop.
- Nombre sugerido: `feature/tema-corto`
- Vida corta, idealmente 1 a 2 días.
- Al terminar, Pull Request a develop con squash.

### hotfix/*
- Se crean desde main para correcciones críticas.
- Merge a main y luego back-merge a develop para no perder cambios.

---

## Flujo operativo recomendado
1. Crear develop desde main.
2. Trabajar cada tarea en feature/*.
3. Abrir PR de feature/* a develop.
4. Ejecutar CI y aprobar PR.
5. Merge a develop.
6. Cuando haya corte de entrega, PR de develop a main.
7. Tag en main y despliegue.

---

## Comandos base

```bash
# Crear develop y publicarla
git checkout main
git pull origin main
git checkout -b develop
git push -u origin develop

# Crear una feature
git checkout develop
git pull origin develop
git checkout -b feature/ci-cd-base
git push -u origin feature/ci-cd-base

# Integrar hotfix
git checkout main
git checkout -b hotfix/fix-health-check
git push -u origin hotfix/fix-health-check
```

## Sugerencia de código para validar nombre de rama en CI

Job liviano en GitHub Actions:

```yaml
name: branch-name-check
on: pull_request:
jobs:
  validate-branch:
    runs-on: ubuntu-latest
    steps:
      - name: Validate source branch
        run: |
          BRANCH="${{ github.head_ref }}"
          echo "Branch: $BRANCH"
          echo "BRANCH" | grep -E "(feature|hotfix|chore|docs)/[a-z0-9.-]+"
```

Esto evita PRs con nombres ambiguos y fuerza disciplina de flujo.

---

## Sugerencia de política de merge
- `feature/*` → `develop` con squash merge.
- `develop` → `main` con merge commit para preservar hito de entrega.
- Prohibir commits directos a main y develop.

---

## Mapa de hitos del examen usando esta estrategia
- `feature/ci-cd-base`
- `feature/k8s-core-manifests`
- `feature/observability-k8s`
- `feature/terraform-foundation`
- `feature/helm-chart`
- `feature/docs-evidencias-finales`
- **PR final**: develop → main

---

## Criterio de aceptación de esta estrategia
- No hay commits directos a main.
- Todo cambio llega por PR con evidencia.
- CI corre en cada PR.
- Se puede reconstruir historial por hitos de entrega sin ambigüedad.
